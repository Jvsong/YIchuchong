# 易趣宠访客端移动适配与云服务器发布设计

**日期：** 2026-06-21

**项目：** 易趣宠（Epet / Yiquchong）

**范围：** 所有访客可见页面；不改造 `/admin` 后台界面

**部署目标：** 阿里云 ECS `8.138.31.44`，目录 `/www/wwwroot/yiquchong`

## 1. 目标

在不改变现有主题颜色、业务功能、内容数据和技术栈的前提下，为全部访客页面建立完整的响应式体验，并将验证通过的版本发布到现有阿里云服务器。

完成后的访客端应满足：

- 在 375px、390px、430px 和 768px 宽度下无非预期横向滚动；
- 导航、按钮、筛选器和表单适合触控操作；
- 标题、正文、产品图片与照片在窄屏上具有正确的层级、换行和裁切；
- 首页、产品目录、产品详情、定位器、百科、资讯、服务和 AI 养宠页面均可正常使用；
- 桌面版视觉和交互不发生明显回退；
- 线上已有环境变量、可编辑内容和上传资源得到保留。

## 2. 明确不在范围内的内容

- 不修改 `/admin` 后台管理界面的移动布局；
- 不更换 Next.js、React 或现有 CSS 技术方案；
- 不改变品牌主题色和主要视觉语言；
- 不新增业务功能、支付流程或新的后台数据结构；
- 不替换现有宠物、产品和资讯内容；
- 不迁移服务器、域名、数据库或对象存储。

## 3. 当前状态与主要问题

项目使用 Next.js 14 App Router、React 18、TypeScript 和单一全局样式文件 `src/app/globals.css`。现有样式已包含 1120px、980px 和 640px 附近的断点，但响应式规则分散在多轮页面设计样式之后，复杂页面仍容易表现为桌面布局的压缩版本。

重点风险包括：

- 导航在窄屏下的展开层级、滚动锁定和操作密度不完整；
- 大型 Hero、浮动说明卡和产品图片在手机端可能相互遮挡；
- 产品目录、产品详情和定位器页面包含复杂网格、缩略图、规格行和筛选器；
- 新闻、百科筛选项在窄屏下容易挤压或产生横向溢出；
- 表单控件和操作按钮需要满足触控尺寸，并避免 iOS 输入时自动缩放；
- 页脚、订阅表单和多列信息需要重新排序，而非等比例压缩；
- `globals.css` 中存在多组后置响应式规则，需要控制覆盖顺序，避免桌面回归。

## 4. 采用方案

采用“完整响应式优化”方案：继续使用现有组件和 CSS 系统，通过共享基础规则、复用组件规则和页面专属规则三层完成改造。必要时允许对组件 JSX 做小幅结构调整，但不重写页面，也不建立独立的移动端代码分支。

未采用的方案：

- 仅补充少量 CSS：改动较小，但无法改善复杂筛选器、产品画廊和导航交互；
- 建立独立移动版：效果可控，但会带来两套界面和更高的长期维护成本。

## 5. 响应式架构

### 5.1 共享基础层

在 `src/app/globals.css` 中统一：

- 容器边距和最大宽度；
- 流式字号和标题换行；
- section 的移动端垂直节奏；
- 图片 `object-fit`、宽高比和最大宽度；
- 按钮、输入框和可点击区域的最小触控尺寸；
- 长文本、标签和网格子元素的 `min-width: 0`；
- 安全区域内边距和非预期横向溢出防护；
- `prefers-reduced-motion` 下的动画降级。

### 5.2 复用组件层

优先改造以下组件：

- `Header`：汉堡菜单、纵向菜单、移动端操作区和页面滚动控制；
- `Footer`：多列信息重排、订阅表单纵向布局；
- `PageHero`：文案、主图、浮动信息卡的移动顺序与裁切；
- `Cards`：卡片媒体比例、正文节奏和底部操作对齐；
- `DevicesCatalog`：筛选、结果信息、产品网格和比较区；
- `WikiExplorer`、`NewsExplorer`：搜索、分段筛选和标签滚动；
- `AiCareDemo`、`ServiceLeadForm`：单列字段、触控控件、结果卡片和错误状态。

### 5.3 页面专属层

对以下访客路由逐页验证和补充规则：

- `/`
- `/devices`、`/devices/[slug]`
- `/locator`
- `/wiki`、`/wiki/[slug]`
- `/news`、`/news/[id]`
- `/boarding`、`/walking`、`/partners`
- `/ai-care`、`/advice`
- `/fun`、`/about`

## 6. 断点与布局行为

断点采用内容驱动策略，而不是针对单一设备：

- `≤1120px`：压缩大型展示和多列产品区域；
- `≤980px`：双栏转单栏，启用移动导航，降低复杂网格列数；
- `≤640px`：进入手机单列布局，调整留白、字号、媒体高度和按钮布局；
- `≤430px`：处理窄屏上的长标题、标签、筛选器、并排按钮和浮动面板。

验收时额外检查 1024px 和 1440px，确保平板横屏和桌面端没有回归。

## 7. 组件交互设计

### 7.1 导航

- 980px 以下显示汉堡按钮；
- 展开菜单为清晰的纵向导航，不与主要操作重叠；
- 保留语言切换和重要入口；
- 菜单展开时防止底层页面误滚动；
- 点击导航项、页面切换或按 Escape 后关闭菜单；
- 保留可见焦点和正确的 `aria-expanded` 状态。

### 7.2 Hero 与产品展示

- 移动端按“标题与说明 → 主要行动 → 产品或场景图片 → 补充信息”排序；
- 浮动卡片不得遮挡产品主体或关键宠物部位；
- 主要图片使用稳定宽高比，避免布局跳动；
- 并排按钮在窄屏下允许纵向排列并占满可用宽度。

### 7.3 产品目录与详情

- 目录筛选器使用紧凑布局；超出宽度时允许有明确边界的横向滚动或折叠；
- 产品卡片依次展示图片、类别、名称、卖点、状态和操作；
- 详情页缩略图在手机端横向滚动，主图保持完整可见；
- 规格行和能力列表转为单列、键值堆叠或可读卡片；
- 主要咨询操作在窄屏上全宽显示。

### 7.4 内容、筛选与表单

- 百科和新闻搜索框全宽；筛选标签可滚动且不挤压文字；
- 文章正文采用舒适行宽与行高，相关推荐在手机端单列；
- 输入框字体不小于 16px，降低 iPhone 自动缩放风险；
- 控件和按钮的有效触控高度约为 44px 或以上；
- 错误、加载、成功内容在窄屏下不溢出。

### 7.5 页脚

- 服务入口、品牌信息和导航分组按阅读顺序纵向排列；
- 订阅表单改为单列；
- 长链接和版权信息允许自然换行；
- 底部留白考虑移动设备安全区域。

## 8. 数据流与兼容性

响应式改造不改变内容数据流：服务端组件继续通过 `src/lib/content.ts` 读取可编辑内容，客户端组件继续通过 props 接收必要数据。图片路径、宠物名称映射、后台存储格式、API 请求和国际化结构保持不变。

如果组件结构调整涉及客户端状态，只允许增加与界面行为直接相关的状态，例如移动菜单开关；不得复制服务端内容读取逻辑或引入新的全局状态库。

## 9. 可访问性与性能

- 所有图像保留有意义的替代文本；
- 菜单、筛选器、表单和按钮可通过键盘操作；
- 保留明显的焦点状态和足够的颜色对比；
- 动画仅使用 `transform` 和 `opacity`，并尊重减少动态效果偏好；
- 不新增不必要的 UI 依赖；
- 优先使用 CSS 完成布局，避免基于窗口宽度的重复 React 渲染；
- 确保图片不会因手机端规则而请求额外的超大资源。

## 10. 验证方案

### 10.1 自动验证

执行：

```bash
npm run validate:content
npm test
npm run lint
npm run build
```

### 10.2 视觉与交互验证

对全部访客路由按以下宽度检查：

- 375px
- 390px
- 430px
- 768px
- 1024px
- 1440px

检查项：

- 页面无非预期横向滚动；
- 导航可展开、关闭并正确跳转；
- 标题无异常截断，正文无需横向阅读；
- 图片不变形、不过度裁切且不遮挡信息；
- 筛选器、产品画廊、表单和主要按钮可操作；
- 浏览器控制台没有新增的运行时错误；
- 桌面端主题颜色、布局和主要操作保持一致。

## 11. 发布设计

沿用现有架构：

```text
用户 → Nginx → Next.js（PM2）
                    ↓
             线上内容与上传资源
```

发布顺序：

1. 通过 SSH 读取服务器当前 Node、PM2、Nginx、Git 和项目状态；
2. 备份 `/www/wwwroot/yiquchong` 中的环境文件、内容存储、上传资源及当前可运行版本；
3. 在本地完成验证、提交并推送到原 GitHub 仓库 `Jvsong/YIchuchong`；
4. 在服务器拉取指定提交，使用锁文件安装依赖并执行生产构建；
5. 使用现有 PM2 进程进行平滑重启；
6. 从服务器本机和公网分别验证首页、关键访客路由、静态资源、API、PM2 与 Nginx；
7. 发现构建、启动、数据或公网访问异常时，恢复备份版本并重启原进程。

部署操作不得打印或提交环境变量、私钥、Cookie 或 API 密钥。

## 12. 完成标准

只有在以下条件同时满足时，工作才视为完成：

- 全部访客页面通过规定宽度的响应式检查；
- 后台页面和内容数据未被响应式改造破坏；
- 内容校验、测试、Lint 和生产构建均通过；
- 修改已提交并推送到原 GitHub 仓库；
- 云服务器成功运行新提交；
- 公网关键页面返回成功响应并可正常操作；
- 线上已有数据、上传资源和环境变量仍然存在；
- 已记录部署提交与回滚点。
